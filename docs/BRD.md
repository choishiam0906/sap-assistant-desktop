# BRD
# SAP Assistant Desktop Platform

## 1. 비즈니스 목적

SAP Assistant Desktop Platform의 목적은 SAP 실무자가 질문, 점검, 유지보수, 온보딩에 쓰는 시간을 줄이고, 동시에 민감 소스 유출 위험을 낮추는 것입니다.

이 제품은 단순 FAQ 챗봇이 아니라 아래 2개 목표를 동시에 달성해야 합니다.

1. SAP 운영 효율화
2. 민감한 내부 자산을 보호하는 안전한 AI 활용

---

## 2. 사업 배경

대부분의 SAP 조직은 다음 현실을 갖고 있습니다.

- 운영 노하우가 사람에게 묶여 있습니다.
- 신규 인력은 T-code, 절차, 덤프 대응, CTS 흐름을 익히는 데 시간이 오래 걸립니다.
- 내부 CBO 소스는 보안 때문에 AI 활용이 제한됩니다.
- 최신 기술 영역인 BTP, RAP, CAP, Integration Suite는 기존 운영 문서만으로 커버되지 않습니다.

이 제품은 SAP 실무자가 “질문할 수 있는 안전한 데스크톱 조수”를 갖게 만드는 것이 핵심 비즈니스 목적입니다.

---

## 3. 목표 사용자와 가치

| 사용자 | 비즈니스 가치 |
|--------|--------------|
| SAP 운영자 / Basis | 장애 대응과 절차 확인 속도 향상 |
| SAP 현업 / Key User | 업무 문의 응답 속도 향상, 운영팀 의존도 감소 |
| ABAP / CBO 담당자 | 내부 소스를 외부 유출 없이 분석 가능 |
| PI / Integration 담당자 | 인터페이스 이슈 대응 지식 통합 |
| BTP / RAP / CAP 개발자 | 최신 표준 기술 질의와 설계 참고 지원 |
| 관리자 / 팀장 | 지식 표준화, 온보딩 비용 절감, 보안 통제 강화 |

---

## 4. 비즈니스 요구사항

### BR-1. 데스크톱 중심 배포
- 제품은 데스크톱 앱 형태로 제공되어야 한다.
- 사용자는 별도 웹 서비스 의존 없이 핵심 기능을 사용할 수 있어야 한다.

### BR-2. 보안 모드 기반 AI 활용
- 조직은 어떤 질문이 외부 모델로 나갈 수 있는지 통제할 수 있어야 한다.
- 기밀 데이터는 기본적으로 local-only 처리되어야 한다.

### BR-3. 역할별 SAP 업무 지원
- 단일 챗봇이 아니라 역할별 업무 시나리오를 지원해야 한다.
- 운영, 현업, CBO, PI/Integration, BTP/RAP/CAP 영역을 구분해야 한다.

### BR-4. 내부 지식과 표준 지식 분리
- 내부 자산과 공개 SAP 레퍼런스는 물리적/논리적으로 분리되어야 한다.
- 사용자는 어떤 데이터가 답변에 사용됐는지 이해할 수 있어야 한다.

### BR-5. 감사 가능성
- 세션별로 질문, 사용 모드, 외부 전송 여부, 참고 근거가 기록되어야 한다.

### BR-6. 단계적 기업 확장
- 초기에는 개인/팀 단위 데스크톱 사용을 지원하고,
- 이후 팀 정책, 중앙 동기화, 엔터프라이즈 게이트웨이로 확장 가능해야 한다.

---

## 5. 비즈니스 제약사항

| 항목 | 제약사항 |
|------|---------|
| 보안 | 내부 CBO 소스는 기본 외부 전송 금지 |
| 정책 | 기업별 AI 허용 정책이 다르므로 모드 기반 제어가 필수 |
| 예산 | 서버 중심 제품보다 Desktop-first로 운영 비용을 줄여야 함 |
| 도입성 | 개발자뿐 아니라 운영자와 현업도 쓸 수 있어야 함 |
| 데이터 | SAP 자격증명 자체를 제품에 저장하지 않는 방향을 우선 |

---

## 6. 기대 효과

### 운영 효율
- T-code, 절차, 장애 대응 흐름 탐색 시간 단축
- 반복 문의와 운영팀의 설명 비용 절감

### 지식 자산화
- 개인 종속 노하우를 구조화된 지식으로 전환
- 역할별 표준 대응 흐름 정리

### 보안 친화적 AI 도입
- “아무 데나 붙여넣는 AI 사용”을 줄이고
- 정책이 있는 승인 가능한 AI 활용으로 전환

### 온보딩 가속
- 신규 운영자와 신규 개발자의 학습 곡선 완화

## 7. 성공 지표

| 지표 | 설명 | 초기 목표 |
|------|------|----------|
| 주간 활성 사용자 수 | 실제 사용 인원 | 10명 이상 |
| 첫 질문 완료 시간 | 실행 후 첫 질문까지 걸리는 시간 | 3분 이내 |
| 운영 질의 해결 시간 | 절차/T-code 탐색 시간 | 30% 단축 |
| CBO local-only 사용 비율 | 민감 질의 중 local-only 처리 비중 | 90% 이상 |
| 예외 전송 건수 | 승인 없는 외부 전송 건수 | 0건 |
| 온보딩 보조 사용률 | 신규 인력의 주간 사용 비율 | 70% 이상 |

## 8. ROI 관점

### 정량 가치
- 운영자가 매일 반복하는 탐색 시간을 줄인다.
- 신규 인력 교육에 필요한 숙련자 시간을 줄인다.
- CBO 검토와 운영 문의 대응 시간을 줄인다.

### 정성 가치
- 보안에 대한 불안 없이 AI를 사용할 수 있다.
- 역할별로 같은 앱을 쓰므로 조직 확산이 쉽다.
- 내부 지식과 공개 지식을 하나의 경험으로 통합할 수 있다.

---

## 9. 도입 전략

### Stage 1
- 개인 또는 소규모 SAP 운영팀 대상 Desktop 배포
- Secure Local Mode 중심 사용

### Stage 2
- Ops, CBO, BTP pack 중심으로 역할 확장
- Reference Mode와 Hybrid Approved Mode 도입

### Stage 3
- PI/Integration, 현업 지원 확대
- 팀 정책과 중앙 관리 기능 도입

---

## 10. 현재 저장소와의 관계

현재 `sap-ops-bot` 저장소는 이미 Desktop, Backend, Frontend 기반을 갖고 있습니다.

다만 제품 관점에서 보면 지금 필요한 것은 기능 추가보다 아래 정렬입니다.

- 제품 포지셔닝 재정의
- 보안 경계 정의
- 역할별 도메인 분리
- Desktop-first 운영 원칙 확정

이번 문서 개편은 그 정렬 작업의 시작점입니다.
